# 故障分级标准 SOP

## 概述
本文档为联通数科云管平台（MSP）生产环境故障分级的唯一权威标准，定义 P0-P3 四级故障的判定条件、响应时间、升级路径、通知范围与复盘要求。所有运维人员、值班人员及模块负责人在故障发生后须首先参照本标准完成级别判定，再依据《应急预案 SOP》启动对应响应流程。dispatch 网关（9091）、cspm 服务、adapter 适配层构成三层核心链路，任一层故障均须按本标准分级处置。

## 角色与职责

| 角色 | 职责 |
|------|------|
| 值班运维工程师 | 首接告警，5 分钟内完成初步判定并记录，执行 P2/P3 处置 |
| 运维负责人 | 审核故障级别，协调跨模块资源，主导 P0/P1 应急指挥 |
| 模块负责人（dispatch/cspm/adapter） | 提供模块级根因分析，执行专项修复方案 |
| 技术总监/VP | P0 级别决策审批，对外沟通与资源调度 |
| SRE 工程师 | 监控告警规则维护，复盘报告质量审核 |

## 故障级别定义

### P0 — 致命
核心服务完全不可用，影响全部用户或核心链路中断。典型场景：dispatch 网关宕机导致 MSP 平台不可访问；Nacos 集群（namespace: prod, daa3b24c-c5b0-4524-8322-58910e7bb739）不可用导致服务发现瘫痪；K8s msp-prod 命名空间大规模 Pod 异常退出。

- **响应时间**：值班人员 5 分钟内响应，15 分钟内启动处置
- **升级路径**：值班人员 → 运维负责人 → 技术总监 → VP
- **通知范围**：全员告警（钉钉群 + 电话），同步客户侧接口人
- **复盘要求**：48 小时内完成复盘报告，须含时间线、根因、改进措施，组织全员评审

### P1 — 严重
核心功能受损，影响部分用户或某云厂商操作不可用。典型场景：某 adapter 宕机导致天翼云/联通云操作不可用；Kafka 消息积压导致异步任务延迟；XXL-JOB（8013 端口）调度任务批量失败。

- **响应时间**：值班人员 10 分钟内响应，30 分钟内启动处置
- **升级路径**：值班人员 → 运维负责人 → 模块负责人
- **通知范围**：运维群钉钉告警，通知受影响项目组
- **复盘要求**：3 个工作日内完成复盘报告

### P2 — 一般
非核心功能异常，存在 workaround。典型场景：报表导出失败但可手动查询；CMDB 资源同步延迟；计费系统（EAV 模型）数据对账异常。

- **响应时间**：值班人员 30 分钟内响应，2 小时内启动处置
- **升级路径**：值班人员 → 模块负责人
- **通知范围**：运维群钉钉告警
- **复盘要求**：周会回顾，纳入故障统计

### P3 — 轻微
不影响业务，需后续修复。典型场景：日志级别过高导致磁盘占用快；Pod 偶发 OOMKill 但 K8s 自动恢复。

- **响应时间**：值班人员当班内响应，次日处理
- **升级路径**：值班人员 → 工单系统
- **通知范围**：工单记录
- **复盘要求**：月度汇总统计

### 分级判定流程
告警触发 → 值班人员初步评估影响范围 → 查询受影响服务/云厂商/项目 → 对照级别定义判定 P0-P3 → 记录判定依据 → 按对应级别启动响应

### 各级别典型场景速查

**P0 典型场景**：
- dispatch 网关全部 Pod 不可用，MSP 平台完全不可访问
- Nacos 集群宕机，所有服务注册发现失败，大面积服务不可用
- K8s msp-prod 命名空间 etcd 异常，Pod 大规模异常重建
- 多云厂商 adapter 同时故障，跨云纳管核心功能全部中断
- PostgreSQL 主库宕机且备库未同步，结构化数据全部丢失

**P1 典型场景**：
- 单个云厂商 adapter 宕机（如联通云 adapter 不可用），影响该云厂商所有操作
- Kafka 消息积压 > 10 万条，计费流水同步延迟 > 30 分钟
- XXL-JOB 调度中心（8013 端口）异常，VM 到 MongoDB 数据同步任务全部失败
- VictoriaMetrics 查询接口响应时间 P99 > 10s，监控大盘全部超时
- AICP 平台 GPU 节点故障，训练任务批量失败

**P2 典型场景**：
- 报表导出功能失败（export_configs 配置异常），用户可手动通过查询接口获取数据
- CMDB 资源同步延迟 > 2 小时，资源列表数据滞后但不影响操作
- 计费系统 EAV 竖表数据对账差异 < 0.1%，不影响月度账单生成
- 单个政务云项目（如烟台）VPN 断连，其他项目正常

**P3 典型场景**：
- Pod 偶发 OOMKill 但 K8s 自动重启恢复，无业务影响
- 日志级别 DEBUG 未及时调整，磁盘占用增长偏快但未达告警阈值
- 单个 ConfigMap 更新后 Pod 未自动 reload（需手动触发）
- Grafana 大盘偶发加载缓慢，刷新后恢复

### 级别边界判定规则

实际排障中常遇到难以判定的边界场景，以下为判定规则：

1. **单 Pod 故障 vs 多 Pod 故障**：单 Pod 故障且 K8s 自动恢复为 P3；单 Pod 故障且未自动恢复、影响功能为 P2；多 Pod 同时故障影响核心功能为 P1；多 Pod 故障导致核心链路中断为 P0。

2. **单云厂商故障 vs 多云厂商故障**：单云厂商 adapter 故障为 P1；两个及以上云厂商 adapter 同时故障为 P0（跨云核心功能受影响）。

3. **政务云特殊判定**：政务云项目（湖南/西安/大连/烟台）故障，因涉及政务合规要求，同等技术影响上调一级处理。例如：普通环境 P2 级别的报表延迟，在政务云环境下按 P1 处理。

4. **监控数据缺失 vs 服务不可用**：VictoriaMetrics 采集中断导致监控数据缺失为 P2（不影响业务但影响可观测性）；VictoriaMetrics 查询接口完全不可用为 P1（影响运维排障能力）。

5. **非工作时间 vs 工作时间**：故障级别判定不以时间为转移，但 P2/P3 在非工作时间可延迟至次日处理，P0/P1 须 7x24 立即响应。

## 真实案例

**时间**：2025 年 10 月 29 日
**事件**：联通 MSP 平台 dispatch 网关因 Nginx upstream 配置中某 adapter 后端地址填写错误，导致全平台接口返回 502。

**时间线**：
- 14:02 — Prometheus 告警触发，dispatch 网关 HTTP 5xx 速率飙升至 100%
- 14:03 — 值班人员电话收到告警，5 分钟内响应
- 14:05 — 确认全平台不可访问，判定 P0 级别
- 14:06 — 值班人员上报运维负责人，同时启动应急小组
- 14:08 — 定位为最近一次 Helm upgrade 引入的 Nginx 配置变更
- 14:10 — 执行 `helm rollback dispatch <revision> -n msp-prod` 回滚配置
- 14:11 — 平台恢复访问，监控指标回归正常
- 14:30 — 确认全链路功能正常，解除应急状态
- 次日 14:00 前 — 完成 48 小时复盘报告

**经验教训**：
1. Nginx 配置变更必须在 CI/CD 流水线中增加 `nginx -t` 预校验步骤
2. Helm upgrade 前须执行 `helm template` 渲染校验 values.yaml 正确性
3. P0 级别电话告警链路有效，从告警到响应仅 3 分钟

## 检查清单
- [ ] 故障级别是否已明确判定并记录判定依据
- [ ] 响应时间是否符合该级别 SLA 要求
- [ ] 升级路径是否按标准执行（未越级、未遗漏）
- [ ] 通知范围是否覆盖所有相关方（含客户侧接口人）
- [ ] 复盘报告是否在规定时间内完成并归档
- [ ] 改进措施是否已录入跟踪系统并分配负责人
- [ ] 故障统计是否已更新至月度报表

## 注意事项
1. 故障级别判定以实际影响为准，不以触发告警数量为准。单个告警若指向核心链路中断，仍应判定 P0。
2. 级别可在处置过程中调整（升级或降级），但每次调整须记录原因和时间点。
3. P0/P1 故障禁止在未完成根因分析前直接关闭，须留存完整证据链（日志截图、Prometheus 指标导出）。
4. 政务云项目（湖南/西安/大连/烟台）的故障级别判定需额外考虑政务侧合规影响，同等技术影响可能上调一级。
5. 多云厂商同时故障时，若影响单一云厂商则按 P1 判定；若影响跨云纳管核心功能则按 P0 判定。

## 工具与资源
- **Prometheus + Grafana**：告警触发与指标可视化，dispatch 网关 5xx 速率告警规则
- **钉钉告警机器人**：Webhook 推送 P0-P2 告警，P0 同时触发电话通知
- **kubectl**：`kubectl -n msp-prod get pods -o wide` 查看 Pod 状态与节点分布
- **Helm CLI**：`helm history <release> -n msp-prod` 查看变更历史辅助根因定位
- **Nacos 控制台**：namespace prod（daa3b24c-c5b0-4524-8322-58910e7bb739）服务注册状态检查
- **故障复盘模板**：Confluence 上的标准复盘报告模板（含时间线表格、根因分析框架、改进措施跟踪表）
